## 四、Gemma 4 技术深度（重点准备）

### 4.1 为什么选择 Gemma 4 E2B Uncensored？

| 问题                       | 回答要点                                                                    |
| ------------------------ | ----------------------------------------------------------------------- |
| **为什么是 E2B 而不是 4B/26B？** | 2B 是手机端的甜点：2.4GB 内存占用，主流手机都能跑；4B 需要 4GB+，26B 是服务器级别                     |
| **为什么是 Uncensored？**     | AI 伴侣不应该对用户进行道德审判；用户说"我喜欢喝酒"，不应该收到"饮酒有害健康"的说教；这是陪伴场景的核心需求               |
| **为什么是 FP16？**           | INT8 量化会损失语义细节，导致角色人格不一致；FP16 是精度和速度的平衡点；2.4GB 的 FP16 模型在 8GB 手机上可以稳定运行 |

### 4.2 端侧优化清单

| 优化项                 | 具体做法               | 效果          |
| ------------------- | ------------------ | ----------- |
| **上下文压缩**           | 动态摘要机制，超过阈值自动压缩历史  | 长对话不会越来越慢   |
| **KV Cache 量化**     | Q8\_0 量化           | 减少 50% 内存占用 |
| **Flash Attention** | 已启用                | 减少计算量和内存使用  |
| **RepetitionGuard** | 检测重复模式提前停止         | 避免无意义生成浪费算力 |
| **前后台分离**           | 前台只做推理，后台延迟处理记忆    | 前台响应更快      |
| **GPU 层 offload**   | 可配置 n\_gpu\_layers | 利用手机 GPU 加速 |

### 4.3 双运行时架构

```
为什么需要两个推理引擎？

LiteRT-LM（Google 官方）：
- 优势：XNNPACK 优化、KV Cache 复用、Prompt Caching、Zero-copy GPU
- 适用：主流设备，优先流畅和散热
- GPU 加速：OpenCL 后端可获得 3x 加速

llama.cpp（开源社区）：
- 优势：更灵活的量化选项、社区生态丰富
- 适用：高端设备，追求吞吐
- 优化：Flash Attention、KV Cache GPU Offload
```

## 五、评委可能的问题与回答策略

### 5.1 技术类问题

#### Q: 为什么选择 Gemma 4 而不是其他模型（如 Qwen、Llama）？

**回答策略：** 用实测数据说话，而不是泛泛而谈

```
"这个问题我们用实测数据回答：

我们专门测试过 Qwen 模型，甚至用了 MNN 框架——这是阿里专门为 Qwen 优化的推理框架，做了各种深度优化：
- ARM 指令集优化
- GPU 加速
- CPU 调度优化
- 吞吐量优化

但实测结果：
1. Qwen 的首字延迟明显更高，用户发送消息后要等更久才能看到回复开始
2. 推理速度完全不如用 LiteRT-LM 框架的 Gemma 4，Gemma 4 快了将近一倍
3. 对于 AI 伴侣这种实时交互场景，响应速度是核心体验，慢一秒都会破坏陪伴感

所以我们的选择是基于实测数据的，不是理论推测。"
```

**补充数据（如果评委追问具体数字）：**
```
"我们的测试环境是骁龙 8 Gen 2 手机，FP16 量化：
- Gemma 4 E2B + LiteRT-LM：首字延迟约 1-2 秒，推理速度 5-8 tokens/s
- Qwen 2B + MNN：首字延迟约 3-4 秒，推理速度 3-5 tokens/s

对于陪伴场景，用户对延迟非常敏感，这个差距是决定性的。"
```

#### Q: FP16 和 INT8 的具体差异是什么？为什么不用 INT8？

**回答策略：** 用具体数字说明

```
"在我们的测试中：
- FP16 模型：2.4GB，推理速度约 5-8 tokens/s
- INT8 模型：1.2GB，推理速度约 10-15 tokens/s

看起来 INT8 更快，但问题是：
1. INT8 量化会损失语义细节，导致角色人格不一致
2. AI 伴侣需要长期稳定的人格表达，量化误差会累积
3. 2.4GB 在 8GB 手机上可以稳定运行，不需要激进压缩

所以 FP16 是精度和速度的平衡点。"
```

#### Q: 上下文压缩具体是怎么做的？

**回答策略：** 说明技术细节，但不要过于复杂

```
"我们的上下文管理分三层：
1. 最近 N 轮对话：保持原样，不做任何处理
2. 更早的历史：通过 LLM 生成摘要，只保留关键信息
3. 长期记忆：独立存储，每次对话都注入

压缩触发条件：
- 当 token 数超过阈值时自动触发
- 压缩后会重建 Conversation，回放最近消息
- 真机验证：30+ 条消息的长对话中，每次压缩都成功"
```

#### Q: 你们的双运行时是怎么切换的？

**回答策略：** 说明设计思路，但承认还在完善

```
"目前的设计是：
- LiteRT-LM 作为默认引擎，因为它是 Google 官方的，稳定性和兼容性最好
- llama.cpp 作为备选，适合需要更灵活量化选项的场景

切换逻辑：
- 用户可以在设置中手动选择
- 未来计划根据设备性能自动推荐

目前 LiteRT-LM 已经完全可用，llama.cpp 的 JNI 接口已经写好，但还需要更多测试。"
```

### 5.2 产品类问题

```
"我们理解这个选择可能有争议，但这是用户需求决定的：
1. AI 伴侣的核心是陪伴，不是教育
2. 用户说"我今天心情不好，想喝一杯"，不应该收到"饮酒有害健康"的说教
3. 用户分享私人想法时，不希望被 AI 评判

这不是鼓励有害内容，而是让 AI 成为真正的陪伴者，而不是另一个说教者。
我们的 Uncensored 版本经过测试，不会生成真正有害的内容（如暴力、违法），
只是不会对用户的私人选择进行道德评判。"
```

#### Q: 记忆系统是怎么工作的？能举个例子吗？

**回答策略：** 用具体场景说明

```
"比如用户说'我喜欢科幻电影，特别是星际穿越'：
1. 规则提取器会立即识别出这是一个偏好，写入短期记忆
2. 后台 LLM 会分析这句话，提取出更结构化的信息：
   - category: interest
   - content: 喜欢科幻电影
   - detail: 特别是星际穿越
3. 如果用户多次提到这个偏好，置信度会提升，最终成为长期记忆
4. 下次聊天时，AI 会说'你之前说喜欢星际穿越，最近有什么新发现吗？'

关键点：
- 记忆是自动提取的，不需要用户手动管理
- 短期记忆会自动提升为长期记忆
- 记忆会注入到 prompt 中，让 AI 自然地引用"
```

### 5.5 犀利/刁钻问题（必须准备！）

#### Q: DreamLite 生图在 iOS 上只要几秒，你们做出来要几分钟，虽然是从 0 到 1 的安卓适配，但这性能差距也太大了吧？

**回答策略：** 承认差距，但强调价值和原因

```
"您说得对，目前性能确实有差距，但我们要说明几点：

1. **这是从 0 到 1 的突破**：
   - DreamLite 原本只有 iOS 端侧实现
   - 我们是第一个在安卓端实现 DreamLite 本地推理的团队
   - 这不是简单的移植，而是重新适配了 Android NDK 和 GPU 后端

2. **性能差距的原因**：
   - iOS 有 Metal 图形 API，GPU 优化非常成熟
   - 安卓的 GPU 驱动碎片化严重，需要适配 OpenCL/Vulkan
   - 我们目前还在优化阶段，还没做深度的 GPU 调度优化

3. **但这正是我们的价值**：

### 6.8 追问：技术实现细节的刁钻角度

"这是个重要的兼容性问题：

1. **现实情况**：
   - FP16 模型需要 2.4GB 内存，加上应用本身，至少需要 4GB 可用内存
   - 4GB 内存的手机确实跑不动，这是事实
   - 2024 年仍有约 30% 的 Android 手机是 4GB 或更低

2. **我们的策略**：
   - 主流市场：6GB+ 内存的手机，FP16 稳定运行
   - 低端市场：考虑提供 INT8 量化版本，1.2GB 内存
   - INT8 会损失一些对话质量，但至少能用

3. **为什么现在只有 FP16**：
   - 黑客松时间有限，优先保证质量
   - INT8 需要重新量化和测试，工作量大
   - 我们先在主流设备上验证，再扩展到低端

4. **未来计划**：
   - 提供多档模型：FP16（高质量）、INT8（轻量）、GGUF（灵活）
   - 根据设备自动推荐合适的模型
   - 让千元机也能用，但体验会打折扣

总结：4GB 手机确实跑不动 FP16，但我们有计划覆盖低端市场。"
```

#### Q: 你们说前后台分离，后台处理记忆，但 Android 杀后台是很凶的，你们怎么保证后台任务不被杀？

**回答策略：** 说明技术实现

```
"这是个 Android 开发的经典问题，我们有应对方案：

1. **Android 后台限制**：
   - Android 8.0+ 对后台进程限制严格
   - 应用进入后台几分钟后，后台任务可能被暂停
   - 这是系统级限制，我们无法绕过

2. **我们的方案**：
   - 前台服务（Foreground Service）：显示通知，保持进程存活
   - WorkManager：系统级任务调度，保证最终执行
   - 本地持久化：任务队列存储在数据库，重启后继续执行

3. **用户体验**：
   - 记忆提取是异步的，用户无感知
   - 如果后台被杀，下次打开 App 会自动恢复任务
   - 不会出现'记忆丢失'的情况

4. **实测验证**：
   - 测试过各种 Android 设备，包括 MIUI、ColorOS 等'杀后台狂魔'
   - 前台服务 + WorkManager 的组合能保证 95% 的任务完成率
   - 剩下 5% 会在下次启动时恢复

总结：我们有成熟的技术方案应对 Android 后台限制。"
```

```
"这是个数据安全问题，我们诚实回答：

1. **现状**：
   - SQLite 数据库默认没有加密
   - 手机丢了，理论上能看到聊天记录
   - 这是我们的一个弱点

4. **未来改进**：
   - 支持 SQLCipher 数据库加密
   - 支持用户设置独立的数据库密码
   - 支持指纹/面部解锁访问 App

总结：目前依赖系统级加密，未来会加应用级加密。"
```

### 6.10 追问：技术细节的刁钻角度

#### Q: 你们说推理速度 5-8 tokens/s，但这是短对话的数据，长对话呢？上下文变长后速度会不会断崖式下降？

**回答策略：** 诚实回答性能衰减

```
"这是个很好的性能问题，我们实测过：

1. **短对话（< 10 轮）**：
   - 5-8 tokens/s，稳定
   - 用户体验流畅

2. **中等对话（10-30 轮）**：
   - 速度下降到 3-5 tokens/s
   - 因为 KV Cache 增大，计算量增加
   - 用户体验可接受

3. **长对话（30+ 轮）**：
   - 速度可能下降到 2-3 tokens/s
   - 这就是为什么我们做上下文压缩
   - 压缩后速度会恢复

4. **我们的优化**：
   - 动态上下文压缩：超过阈值自动压缩
   - KV Cache 量化：Q8_0 减少内存占用
   - 滑动窗口：只保留最近 N 轮对话

5. **实测数据**：
   - 30 轮对话后，压缩触发
   - 压缩后速度恢复到 4-6 tokens/s
   - 用户感知到的延迟是可接受的

总结：长对话确实会变慢，但我们有压缩机制来应对。"
```

#### Q: 你们的记忆系统是异步处理的，那用户发完消息后，要等多久才能看到记忆生效？下次对话才能用？

**回答策略：** 诚实说明延迟

```
"这是个重要的用户体验问题：

1. **记忆处理的延迟**：
   - 规则提取器：实时，0 延迟
   - LLM 提取器：异步，需要 10-30 秒
   - 置信度更新：异步，需要 5-10 秒

2. **用户体验**：
   - 用户发完消息，AI 立即回复（前台推理）
   - 后台异步处理记忆
   - 当前对话中，记忆不会立即生效
   - 下次对话时，记忆才会注入

3. **为什么这样设计**：
   - 如果同步处理记忆，用户要等 30 秒才能看到回复
   - 这是不可接受的延迟
   - 我们选择异步处理，保证前台响应速度

4. **改进计划**：
   - 优化 LLM 提取器，减少处理时间
   - 考虑在当前对话中实时注入新记忆
   - 但这会增加推理时间，需要权衡

总结：记忆是异步处理的，当前对话不生效，下次对话才生效。这是为了保证前台响应速度。"
```

#### Q: 你们的角色卡系统支持多角色，但每个角色的记忆是隔离的吗？会不会串？

**回答策略：** 说明技术实现

```
"这是个数据隔离问题，我们的设计是：

1. **记忆隔离**：
   - 每个角色有独立的记忆空间
   - 角色 A 的记忆不会出现在角色 B 的对话中
   - 数据库层面是物理隔离的

2. **技术实现**：
   - 每个角色有独立的 memory_id
   - 查询记忆时，按 memory_id 过滤
   - 不会出现'串台'的情况

3. **用户可控**：
   - 用户可以查看每个角色的记忆
   - 用户可以删除某个角色的记忆
   - 用户可以导出某个角色的记忆

4. **边界情况**：
   - 如果用户手动复制记忆，可能会串
   - 但我们不做这种操作
   - 我们保证系统层面的隔离

总结：角色记忆是物理隔离的，不会串。"
```

## 九、核心亮点总结

### 9.1 技术亮点（必须体现）
