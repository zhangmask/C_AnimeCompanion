# CompanionChat 阶段二补齐设计

> 日期：2026-05-13
> 目标：以最小闭环方式补齐阶段二剩余缺口，在不引入第二引擎的前提下，完善摘要、降级、长对话验收与 UI 可操作性验收。

## 1. 背景

当前阶段二主链路已经完成：

- `ContextSettings`、`PromptAssembler`、`ContextManager` 已接入
- `Conversation` 重建与最近消息回放已经在真机上验证成功
- 设置页 `N` 配置入口已生效

但从阶段二清单看，仍有四类缺口没有完全闭环：

- 缺少真实可用的最小摘要器
- 回放失败后的降级路径未明确注入“最近对话摘要/片段”
- 缺少 `30+` 条真机长对话专项验收记录
- 缺少压缩期间 UI 可操作性的专项验收记录

## 2. 本次目标

本次只补齐阶段二，不提前进入阶段四能力。

具体目标：

- 增加一个稳定、可测试、不依赖第二引擎的最小摘要器
- 让回放失败时真正走“摘要注入 + 最近片段补足”的降级路径
- 形成可复现的 `30+` 条真机长对话验收流程
- 形成压缩期间 UI 仍可操作的人工验收记录

## 3. 非目标

- 不接入 `EngineBackedSummaryGenerator`
- 不新增第二个推理引擎生命周期
- 不引入记忆检索、偏好总结或技能切换业务
- 不重构设置页结构

## 4. 方案选择

本次采用“最小闭环补齐”方案。

### 4.1 为什么不用模型摘要

- 阶段二当前缺口的核心是“可用性与验收闭环”，不是摘要质量极限
- 如果此时接入模型摘要，会把问题扩展到双引擎管理、超时、抢占和内存
- 这会明显提升实现风险，并把阶段二拖入阶段四范围

### 4.2 为什么采用规则摘要

- 规则摘要可以稳定控制长度与格式
- 可以直接在单元测试中验证输出长度和降级行为
- 可以满足阶段二“摘要存在且主链路不被阻塞”的工程目标

## 5. 总体方案

```text
ChatViewModel
  -> ContextManager
      -> SummaryGenerator
          -> RuleBasedSummaryGenerator
  -> PromptAssembler
  -> LiteRTLMInferenceEngine
      -> rebuildConversation()
      -> replayMessages()
      -> rebuildConversationWithFallbackContext()
```

核心变化有两处：

1. `SummaryGenerator` 从 `NoOp` 切换为规则摘要实现
2. 回放失败时，不只是恢复到“空历史可发送”，而是显式重建为“摘要注入 + 最近原文片段”的降级会话

## 6. 规则摘要器设计

### 6.1 新增实现

新增 `RuleBasedSummaryGenerator`：

- 输入：被裁剪的历史消息 + `ContextSettings`
- 输出：一段不超过 `summaryMaxChars` 的摘要

### 6.2 生成规则

采用稳定的文本压缩规则：

1. 过滤空消息
2. 将消息转为统一格式：
   - 用户：`用户：{内容}`
   - 助手：`助手：{内容}`
3. 只取每条消息的前若干字符，避免单条过长
4. 按时间顺序拼接
5. 超过 `summaryMaxChars` 时截断，并保留结尾省略标记

### 6.3 设计取舍

- 优点：稳定、可测、无额外引擎依赖
- 缺点：不具备真正语义提炼能力

这次接受该取舍，因为目标是补齐阶段二闭环。

## 7. 降级摘要注入设计

### 7.1 触发条件

只要最近消息回放失败，就进入降级：

- `initialMessages` 不被 LiteRT-LM 接受
- 回放过程抛异常
- 会话创建成功但回放结果不可用

### 7.2 降级内容

降级后新的 `systemPrompt` 由三部分组成：

- 基础 system prompt
- 历史摘要
- 最近片段补足

最近片段补足格式固定为：

```text
最近几轮对话片段：
用户：...
助手：...
```

### 7.3 设计目的

- 即使回放失败，仍尽量保留最近轮次细节
- 保证“回放失败”与“丢失上下文”不是同义词
- 让清单中的降级要求可以被真实验证

## 8. 代码边界

### 8.1 `PromptAssembler`

新增一个用于降级场景的组装入口，负责在 `systemPrompt` 中拼接：

- 用户偏好
- 历史摘要
- 最近片段

避免把降级拼接逻辑散落在 `ViewModel` 或引擎层。

### 8.2 `DefaultContextManager`

继续负责：

- 计算是否压缩
- 生成 `ContextWindow`
- 调用摘要器生成历史摘要

不负责直接重建引擎会话。

### 8.3 `LiteRTLMInferenceEngine`

新增一个更明确的降级重建入口，例如：

```kotlin
suspend fun rebuildConversationWithFallbackContext(systemPrompt: String): Boolean
```

职责：

- 关闭旧会话
- 创建带降级 system prompt 的新会话
- 不做 `initialMessages` 回放

## 9. 测试设计

### 9.1 单元测试

新增或补充以下测试：

- `RuleBasedSummaryGeneratorTest`
  - 返回非空摘要
  - 摘要长度不超过 `summaryMaxChars`
  - 空消息列表返回空字符串
- `PromptAssemblerTest`
  - 降级场景包含“最近几轮对话片段”
  - 片段为空时不拼空标题
- `DefaultContextManagerTest`
  - 使用规则摘要器时，`historySummary` 不再固定为空

### 9.2 编译验证

- `.\gradlew.bat :app:testDebugUnitTest`
- `.\gradlew.bat :app:assembleDebug`

## 10. 真机验收设计

### 10.1 `30+` 条消息专项

目标：

- 连续发送 `30+` 条消息不崩溃
- 至少触发一次压缩
- 压缩后继续追问能得到合理回答

记录内容：

- 当前 `N`
- 压缩触发阈值
- 首次触发压缩时的消息数
- 是否走回放或降级
- 追问结果

### 10.2 UI 可操作性验收

在压缩触发前后人工执行：

- 滑动消息列表
- 点击返回或切换页面
- 再返回聊天页

验收标准：

- 页面不 ANR
- 没有明显卡死
- 用户可以继续操作

## 11. 风险与缓解

### 风险 1：规则摘要太弱

缓解：

- 保留长度限制与最近片段补足
- 本次只把它定义为阶段二闭环实现，不视作最终摘要方案

### 风险 2：降级逻辑和主方案耦合过深

缓解：

- 由 `PromptAssembler` 统一负责拼接
- 引擎层只负责“按给定 prompt 重建会话”

### 风险 3：真机长对话人工测试不稳定

缓解：

- 保持 `N=5`，更早触发压缩
- 先用短消息快速造数据
- 关键结果以日志和截图双重留档

## 12. 结论

本次采用“规则摘要器 + 真实降级摘要注入 + 真机长对话验收 + UI 可操作性验收”的最小闭环补齐方案。

这样可以在不引入第二引擎的情况下，把阶段二剩余缺口补到更完整，并为后续阶段三开发保留稳定基础。
