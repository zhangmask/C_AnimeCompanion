# CompanionChat 阶段三补丁：长期记忆常驻上下文与 FTS 增强设计

> 日期：2026-05-13
> 依据：`2026-05-13-stage3-memory-system-design.md`、`2026-05-13-stage3-memory-system-plan.md`、当前阶段三实现现状
> 目标：在不推翻阶段三已有实现的前提下，让全部长期记忆稳定进入上下文，并提升 FTS 检索的真实召回效果。

## 1. 背景

当前阶段三已经完成了以下主链路：

- 规则提取写入 `memories`
- `MemoryScreen` 最小可用版
- 启动时短期记忆清理与长期提升
- 发送前基于 FTS 检索相关记忆并注入 prompt

但从真机表现看，阶段三仍有两个明显问题：

- 手动写入的长期记忆，尤其是角色设定、关系类记忆，效果不稳定
- 当前 FTS 检索命中率偏低，导致“记忆存在但没有进入上下文”

这说明当前阶段三的主要短板不是“没有记忆注入能力”，而是：

- 长期高价值记忆过度依赖检索
- 当前检索器过于保守，中文问句和关系类问法召回不足

因此，本补丁不重做阶段三，而是在现有架构上补上两个增强层：

- 全部长期记忆常驻上下文
- FTS 检索增强

## 2. 设计目标

本补丁要达成以下结果：

- 所有 `long_term` 记忆在发送前都进入常驻上下文
- 当前问题相关的记忆仍通过 FTS 动态检索补充注入
- 角色设定、关系、长期偏好等长期记忆不再依赖检索命中才生效
- 中文问句的 FTS 命中率明显提升
- 日志中可以明确区分“常驻记忆注入”和“动态记忆检索”

## 3. 范围与非目标

### 3.1 本补丁范围

- 新增长期记忆常驻上下文构造
- 增强 `MemoryRetriever`
- 扩展 `MemoryPromptBuilder`
- 调整 `ChatViewModel` 发送前链路
- 增加对应单测与真机验收项

### 3.2 非目标

- 不在本补丁中引入向量检索
- 不在本补丁中引入新的嵌入模型或向量存储
- 不在本补丁中加入长期记忆条数上限
- 不在本补丁中做长期记忆权重学习

## 4. 关键决策

### 4.1 全部长期记忆进入常驻上下文

原因：

- 用户明确要求长期记忆应稳定生效，尤其是角色设定和关系
- 当前项目中的长期记忆规模预期较小，不需要先加注入上限
- 长期记忆本身就是“跨轮次稳定成立的信息”，适合固定注入

结论：

- 所有 `layer = long_term` 的记忆都进入常驻上下文
- 本补丁不设置常驻记忆条数上限

### 4.2 保留 FTS，但职责从“唯一入口”变为“补充入口”

原因：

- 当前 FTS 虽然召回不足，但已有 Room 表、触发器和查询链路
- 全部放弃 FTS 改向量检索成本过高，不适合作为阶段三补丁
- 常驻长期记忆能兜底稳定信息，FTS 只需负责补充当前问题相关记忆

结论：

- 长期记忆常驻注入
- 动态相关记忆继续通过增强后的 FTS 注入

### 4.3 发送前 prompt 分成“常驻段 + 动态段”

原因：

- 便于区分“稳定背景信息”和“当前问题相关补充”
- 便于日志、调试与后续优化
- 便于未来在不破坏结构的前提下增减注入策略

结论：

- 新增常驻长期记忆段
- 保留动态检索记忆段

## 5. 总体架构

```text
ChatViewModel
  -> MemoryRepository
      -> getPersistentMemories()
      -> retrieveRelevantMemories()
      -> MemoryRetriever
      -> MemoryPromptBuilder
      -> MemoryDao
  -> ContextManager
  -> PromptAssembler
  -> LiteRTLMInferenceEngine
```

职责变化如下：

- `MemoryRepository`
  - 新增获取全部长期记忆的方法
  - 保留动态检索入口

- `MemoryPromptBuilder`
  - 同时负责构造常驻长期记忆段与动态检索记忆段

- `ChatViewModel`
  - 发送前同时构造两段记忆 prompt
  - 日志中分别记录常驻条数与动态命中条数

## 6. Prompt 注入设计

### 6.1 常驻长期记忆段

固定标题：

```text
长期记忆中的关键信息：
- [关系] 小王是用户同事
- [事实] 用户的角色设定是项目负责人
```

规则：

- 只要存在 `long_term` 记忆，就注入该段
- 全部长期记忆进入该段
- 不设置条数上限

### 6.2 动态检索记忆段

固定标题：

```text
从记忆中检索到的与当前对话相关的信息：
- [事实] 用户喜欢 Kotlin
- [偏好] 用户偏好简洁回答
```

规则：

- 只注入当前问题检索命中的结果
- 作为对常驻长期记忆的补充，不替代常驻段

### 6.3 最终组装顺序

`PromptAssembler` 中组装顺序调整为：

1. 基础 system prompt
2. 用户偏好
3. 常驻长期记忆段
4. 动态检索记忆段
5. 历史摘要
6. 最近对话片段（如有）

## 7. FTS 增强设计

### 7.1 当前问题

当前 `MemoryRetriever` 的主要问题有：

- 只取一个最长关键词
- 中文问句没有做归一化
- 弱词、虚词没有过滤
- 关系类问法无法稳定映射到记忆内容

### 7.2 改进方向

#### 7.2.1 多关键词查询

- 不再只取单个最长词
- 提取多个有效关键词
- 对关键词去重
- 使用 `OR` 或等价 FTS 查询表达式组合

#### 7.2.2 弱词过滤

过滤以下类型：

- 纯符号
- 单字符碎片
- 无语义弱词，如“什么”“怎么”“一下”“这个”“那个”等

#### 7.2.3 中文问句归一化

第一版补以下映射：

- `什么关系` / `和我什么关系` -> 保留主体词 + `关系`
- `是谁` -> 保留主体词
- `叫什么` -> `名字`
- `喜欢什么` -> `喜欢`
- `住在哪` -> `住在`

目标不是做完整 NLP，而是让高频记忆问法更容易命中。

### 7.3 查询排序

保留当前排序原则：

- 长期记忆优先
- 再按 `referenceCount` 降序
- 再按 `updatedAt` 降序

## 8. 分类标签调整

当前 `MemoryPromptBuilder` 需要补齐分类映射：

- `fact` -> `事实`
- `preference` -> `偏好`
- `event` -> `事件`
- `relationship` -> `关系`

## 9. 日志与可观测性

发送前新增如下日志：

- `常驻长期记忆注入: count=N`
- `动态记忆检索成功: count=N, query=...`
- `动态记忆检索为空: query=...`

这样能快速判断：

- 是长期记忆没有注入
- 还是动态检索没有命中
- 还是命中后模型没有利用

## 10. 测试与验收映射

### 10.1 后端

- 全部长期记忆能构造成常驻段
- `relationship` 标签显示为 `关系`
- 多关键词查询构造正确
- 中文问句归一化后命中关系类和事实类记忆
- 动态检索命中后引用次数递增

### 10.2 集成

- 手动新增长期关系记忆后，不依赖精确关键词也能稳定影响回答
- 发送问题时日志可同时看到常驻长期记忆注入条数与动态命中条数
- 常驻长期记忆与阶段二上下文管理共存，不破坏现有压缩链路

## 11. 风险与缓解

### 风险 1：长期记忆逐渐增多导致 prompt 变大

缓解：

- 当前先遵循用户要求，不设置数量上限
- 保持常驻段独立封装，未来如需增加排序或限制，不需要重构主链路

### 风险 2：中文问句归一化规则不够全

缓解：

- 第一版只覆盖高频问法
- 保留日志，便于根据真实提问继续补规则

### 风险 3：FTS4 对词组查询行为与预期不一致

缓解：

- 为检索构造器补单测
- 用真机日志和实际问答做集成验证

## 12. 结论

本补丁采用“**全部长期记忆常驻上下文 + 增强 FTS 动态检索**”的方案。

它不推翻阶段三已有实现，而是在现有记忆系统之上补上一个更稳的长期背景层，并提升 FTS 对中文问句和关系类场景的可用性，使阶段三从“功能已接通”提升到“体验可用”。
