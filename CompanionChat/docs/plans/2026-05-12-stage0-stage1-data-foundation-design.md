# CompanionChat 阶段0/阶段1补齐设计

> 日期：2026-05-12
> 范围：严格按 `COMPANIONCHAT_TEST_CHECKLIST.md` 顺序，先补阶段0遗漏的数据层骨架，再做阶段1会话存储迁移与真机验证。

## 1. 目标

- 补齐阶段0中尚未落地的数据层骨架，但只做结构与初始化，不提前实现阶段3/4/5业务功能。
- 完成阶段1的旧 `conversations.json` 向 Room 的迁移闭环。
- 保证 `ChatViewModel` 的会话读取、保存、新建、删除、切换都以 Room 为准。
- 在本轮结束后执行编译、卸载旧 app、安装新 app、推送模型、真机验证。

## 2. 阶段0补齐范围

- 为 `ConversationEntity` 增加 `updatedAt`，与清单字段保持一致。
- 拆分并补齐 DAO：
  - `ConversationDao`
  - `MessageDao`
  - `MemoryDao`
  - `PreferenceDao`
  - `SkillDao`
- 增加阶段0要求的数据表骨架：
  - `Memory`
  - `UserPreference`
  - `Skill`
- 在数据库初始化时创建 `memories_fts` 及其同步触发器。
- 通过 `RoomDatabase.Callback.onCreate` 预填充 4 条内置技能。

## 3. 阶段1实现范围

- 新增独立 `DataMigration` 组件，负责：
  - 检测 `conversations.json`
  - 检测 `.bak` 并跳过重复迁移
  - 解析旧 JSON
  - 导入 `conversations` 与 `messages`
  - 成功后备份原文件
  - 失败时保留原文件并记录日志
- `ChatSessionRepository` 收口为会话读写仓储，不再承载全部迁移解析逻辑。
- `ChatViewModel` 保证以下行为直接或最终落到 Room：
  - 启动加载会话
  - 发送消息后即时保存
  - 新建会话即时入库
  - 删除会话级联删除消息
  - 切换会话与改标题后数据保持一致

## 4. 关键设计决策

### 4.1 数据层先补骨架，不提前做业务

- 阶段0只补表、DAO、数据库回调与初始化，不做记忆页、技能页、偏好总结等业务接线。
- 这样既满足清单，也避免把后续阶段逻辑提前耦合进本轮。

### 4.2 迁移组件独立

- 将旧 JSON 迁移从 `ChatSessionRepository` 中拆出到 `DataMigration`。
- 好处：
  - 迁移职责单一
  - 更容易补测试与日志
  - 后续如果需要“重新导入入口”，也可以直接复用该组件

### 4.3 会话写入以“当前会话全量替换”维持一致性

- 当前阶段继续采用“单会话全量替换消息列表”的方式落库。
- 原因：
  - 现有 UI 状态已是完整会话快照
  - 改动量更小
  - 足以满足阶段1“即时写入”和“删除级联”的清单要求
- 后续如果阶段二需要更细粒度上下文控制，再考虑按消息增量写入。

## 5. 错误处理

- 迁移解析失败：保留原 `conversations.json`，不生成 `.bak`，日志记录异常。
- `.bak` 已存在：视为已迁移过，直接跳过。
- Room 写入失败：页面不崩溃，`ViewModel` 记录错误日志并保留内存态。
- 空数据首次启动：保证至少能正常进入应用，不因无旧数据或空库而崩溃。

## 6. 验证计划

- 本地验证：
  - `:app:assembleDebug`
  - 最近改动文件诊断检查
- 真机验证：
  - 卸载旧包
  - 安装新包
  - 推送模型
  - 注入旧版 `conversations.json`
  - 验证迁移、`.bak`、数据库内容、会话读写、删除级联、重启持久化

- 不实现阶段四的双引擎偏好总结。
- 不实现阶段五的技能管理 UI 与切换逻辑。
