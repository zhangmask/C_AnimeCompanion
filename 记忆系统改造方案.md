# 记忆系统改造方案（增强版 v2）

> 对应审查报告：`记忆系统审查报告.md`
>
> 整合来源：**OpenViking**（链接枚举 + PPR 图检索 + L0/L1/L2 分层 + Agent Experience）、**Hindsight**（多路并行召回 + RRF 融合 + 三层记忆类型）、**mem0**（实体提取 + BM25 归一化评分 + linked_memory_ids 增长控制）、**MetaMem**（元记忆自演化）、**memU**（类型化记忆分类）

---

## 总设计思路

引入 **衰减-强化记忆模型**（艾宾浩斯遗忘曲线）+ **LLM 统一提取** + **类型化链接图** + **PPR 轻量图检索** + **分层上下文注入**。

每条记忆是一个**图节点**，节点间通过 6 种类型化的有向边关联。检索时从用户输入提取实体作为种子，沿边做**轻量 Personalized PageRank（PPR）扩散**，多种子叠加发现桥接记忆，按最终评分排序。

```
  用户说话
     │
     ▼
  ┌──────────────────────────┐
  │  阶段一：离线 LLM 统一提取 │ ← 对话轮结束后触发
  │  (ExtractLoop)            │    输出记忆操作 + 链接（一次调用）
  └──────────┬───────────────┘
             │ 提取结果
             ▼
  ┌──────────────────────────┐
  │  图记忆仓库（SQLite 邻接表） │
  │  ┌─────┐  linked_to  ┌─────┐
  │  │喜欢篮球│──────────→│篮球  │  ← 6 种链接类型
  │  └──┬──┘  belongs_to └─────┘  ← 权重 0~1
  │     │             ↑           ← 行号级精度
  │  ┌──▼──┐  linked_to │
  │  │内向  │────────────┘
  │  └─────┘
  │  每条节点有 strength（0~1）
  │  每日按遗忘曲线衰减
  └──────────┬───────────────┘
             │
             ▼
  ┌──────────────────────────┐
  │  PPR 轻量图检索           │
  │  1. 提取用户消息中的实体    │
  │  2. 匹配实体节点作为种子    │
  │  3. 1-hop PPR 带权扩散    │
  │  4. 多种子桥接叠加分数     │
  │  5. LIMIT 控制 + Token预算 │
  └──────────────────────────┘
```

## 一、图数据模型（增强）

### 1.1 Memory 实体（数据库层）

保留了衰减强度模型，新增 `entityName`（标准化实体名）和 `abstractionLevel`（分层级别）：

```kotlin
data class Memory(
    @PrimaryKey(autoGenerate = true) val id: Long = 0,
    val content: String,
    val category: String,               // profile / preference / event / behavior / knowledge / skill
    val entityName: String? = null,     // 标准化实体名（如 "user_张三"）
    val strength: Double = 0.6,         // 0.0 ~ 1.0，替代 layer
    val abstractionLevel: Int = 2,      // 0=L0摘要 / 1=L1概览 / 2=L2详情
    val source: String,
    val sessionId: String? = null,
    val roleCardId: Long? = null,
    val createdAt: Long,
    val updatedAt: Long,
    val lastAccessedAt: Long = 0
)
```

移除 `layer` 和 `expiresAt` 字段。

### 1.2 链接表（新增 `memory_links` 表）

参考 OpenViking 的 6 种 LinkType 枚举 + mem0 的 linked_memory_ids 增长控制：

```kotlin
@Entity(
    tableName = "memory_links",
    indices = [
        Index("fromId"),
        Index("toId"),
        Index("linkType"),
        Index("fromId", "toId", "linkType", unique = true)
    ]
)
data class MemoryLink(
    @PrimaryKey(autoGenerate = true) val id: Long = 0,
    val fromId: Long,                    // 源记忆 ID
    val toId: Long,                      // 目标记忆 ID
    val linkType: String,                // "related_to" / "belongs_to" / "caused_by"
                                         // / "derived_from" / "contradicts" / "evolved_from"
    val weight: Double = 1.0,            // 0.0 ~ 1.0
    val createdAt: Long,
    val updatedAt: Long
)
```

**链接类型定义（6 种，从 OpenViking 引入）：**

| linkType | 语义 | 示例 | PPR 权重 | 是否继续传播 |
|---|---|---|---|---|
| `related_to` | 一般关联 | 篮球 → 运动 | 0.4 | 否（1跳即止）|
| `belongs_to` | 归属关系 | 喜欢篮球 → 用户 profile | 0.7 | 是（最多3跳）|
| `caused_by` | 因果关系 | 加班 → 疲惫 | 0.5 | 是（最多2跳）|
| `derived_from` | 派生关系 | 周报摘要 → 周报详情 | 0.2 | 否 |
| `contradicts` | 矛盾关系 | 喜欢辣 → 不吃辣 | 0.8 | 否（强制包含）|
| `evolved_from` | 演变关系 | 旧偏好 → 新偏好 | 0.3 | 否 |

### 1.3 实体表（新增，参考 mem0 的 Entity Store）

```kotlin
@Entity(

    val type: String,                    // "person" / "org" / "topic" / "concept"
    val linkedMemoryCount: Int = 1,      // 链接到的记忆数（影响权重衰减）
    val createdAt: Long,
    val updatedAt: Long
)
```

**实体与记忆的多对多关联表：**

```kotlin
@Entity(
    tableName = "memory_entity_map",
    indices = [Index("entityId"), Index("memoryId")]
)
data class MemoryEntityMap(
    @PrimaryKey(autoGenerate = true) val id: Long = 0,
    val entityId: Long,

```

### 1.4 分层上下文（新增，参考 OpenViking L0/L1/L2）

记忆写入时自动生成 3 个层级的文本：

| 层级 | 名称 | Token 预算 | 用途 |
|---|---|---|---|
| L0 | 摘要 | ~50 tokens | 快速相关性判断、FTS 检索引擎 |
| L1 | 概览 | ~200 tokens | 注入 LLM 上下文的主要载体 |
| L2 | 详情 | 全文 | 仅当 LLM 主动请求时读取 |

```kotlin
// Memory 实体新增字段
val l0Summary: String? = null,   // 一句话摘要
val l1Overview: String? = null,  // 包含核心信息的概览
```

L0 用于 FTS 索引和快速匹配；L1 作为上下文注入主体的预算控制单位（默认注入 3~5 条 L1 概览，总计 ~1000 tokens）。

## 二、LLM 统一提取管道（ExtractLoop）

### 2.1 单次 LLM 调用，输出记忆操作 + 链接 + 实体

参考 **OpenViking ExtractLoop** + **mem0 的 8-phase V3** + **Hindsight 的 3 层记忆类型**：

```
对话轮结束（finishStreaming 触发）
    │
    ├── 1. 上下文收集
    │     ├── 已有相关记忆（FTS 检索 + PPR 1-hop）
    │     ├── 已有实体索引
    │     └── 本轮消息 + 最近 K 条历史
    │
    ├── 2. 单次 LLM 调用
    │     输出结构化 JSON：
    │     {
    │       "memories": [
    │         {"op": "add", "category": "preference", "content": "用户喜欢打篮球",
    │          "entityName": "user_张三", "l0Summary": "喜欢篮球", ...},
    │         {"op": "update", "id": 42, "strength": 0.15},
    │         {"op": "delete", "id": 13}
    │       ],
    │       "entities": [
    │         {"name": "张三", "type": "person"},
    │         {"name": "篮球", "type": "topic"}
    │       ],
    │       "links": [
    │         {"fromIdx": 0, "toEntityIdx": 1, "type": "related_to", "weight": 0.8},
    │         {"fromIdx": 0, "toEntityIdx": 0, "type": "belongs_to", "weight": 0.9}
    │       ]
    │     }
    │
    ├── 3. 批量处理
    │     ├── Hash 去重（SHA-256 of normalized content）
    │     ├── 语义去重（嵌入向量 cosine ≥ 0.85 视为重复）
    │     ├── 实体去重合并（标准化名匹配，threshold=0.95）
    │     └── 链接去重（同 from-to-type 取 weight max）
    │
    ├── 4. 持久化（事务性）
    │     ├── INSERT/UPDATE/DELETE memories
    │     ├── INSERT/UPDATE entities（标准化名冲突时合并）
    │     ├── INSERT memory_entity_map
    │     └── INSERT memory_links
    │
    └── 5. 异步后处理
          ├── 嵌入向量化（后台 worker）
          ├── 生成 L0/L1 摘要（LLM 或规则压缩）
          └── 检查矛盾链接（contradicts 类型 → 标记待处理）
```

### 2.3 触发器优化

当前阶段四在用户发送后 30s 静置才触发。改为：

```kotlin
// ChatViewModel.kt 在 finishStreaming 末尾调用
preferenceLearningCoordinator.triggerNow(
    reason = "对话轮结束",
    sessionId = currentSessionId,
    messages = currentMessages
)
```

节流时间缩短为 **15s**（原 30s）。若用户在 15s 内连续对话，等待下一个静置期。

**功耗考虑**：LLM 调用只发生在对话轮结束 + 空闲时，不影响对话流畅性。每天约调用 ~50 次（按活跃用户计算），每次约 ~500 tokens 输入 + ~200 tokens 输出，月均 LLM 成本 ~$0.50。

## 三、PPR 轻量图检索（替代简单 BFS）

### 3.1 从 BFS 升级到 PPR

参考 **OpenViking 的 PPR 搜索增强** + **mem0 的实体提升机制**：

```
Phase 1: 种子获取
  1. 从用户消息中提取实体名（规则提取 + 匹配 memory_entities）
  2. FTS 检索消息原文 → 降序取 top-3
  3. 种子 = 实体匹配的记忆 ∪ FTS 命中的记忆
  4. 每个种子有 base_score（FTS 排名分 / 实体匹配分）

Phase 2: PPR 1-hop 传播
  for each seed in 种子:
    query memory_links WHERE fromId = seed.id OR toId = seed.id
    遍历每条链接：
      - 对端 ID = (link.fromId == seed.id) ? link.toId : link.fromId
      - 按 linkType 查配置表获取传播权重
      - ppr_score = base_score × damping(0.85) × link_weight
      - 如果 weight < 0.3 跳过
      - 累加到对端记忆的 ppr_score

Phase 3: 多种子桥接
  同一记忆被多个种子到达时，分数叠加
  发现桥接文件（非种子但连接了多个种子）→ 提高排序

Phase 4: 分数合并与裁剪
  for each 候选记忆:
    if 在种子中: final_score = 0.7 × base + 0.3 × ppr
    else: final_score = ppr_score
  ppr < 0.05 的丢弃
  LIMIT 5 返回
```

PPR 配置表（SQLite 硬编码常量）：

| linkType | 传播权重 | 是否继续 | 说明 |
|---|---|---|---|
| related_to | 0.4 | 否 | 1跳即止 |
| belongs_to | 0.7 | 是（最多2跳） | 追上下文 |
| caused_by | 0.5 | 是（最多2跳） | 追根因 |
| derived_from | 0.2 | 否 | 溯源，低权重 |
| contradicts | 0.8 | 否 | 矛盾强制包含 |
| evolved_from | 0.3 | 否 | 演变低权重 |

### 3.2 SQL 实现示例

```sql
-- 获取某记忆的所有出边邻接
SELECT ml.*, m.content, m.category, m.strength, m.l0Summary
FROM memory_links ml
JOIN memories m ON m.id = ml.toId
WHERE ml.fromId = :memoryId
  AND ml.weight >= 0.3

UNION ALL

-- 获取某记忆的所有入边邻接
SELECT ml.*, m.content, m.category, m.strength, m.l0Summary
FROM memory_links ml
JOIN memories m ON m.id = ml.fromId
WHERE ml.toId = :memoryId
  AND ml.weight >= 0.3
```

**功耗考虑**：
- PPR 传播限制 1-hop，最多 50 条链接查询
- 纯 SQLite 操作，毫秒级完成
- 种子通常 2~5 个，总计算量约 200~500 行 SQL 扫描
- 不需要 LLM 或向量引擎参与检索阶段

## 四、三层记忆架构

参考 **Hindsight 的 World/Experiences/Mental Models** + **OpenViking 的 Trajectory/Experience** + **memU 的 6 类型分类**：

### 4.1 记忆层级

| 层 | 名称 | 来源 | 类似人类记忆 | 强度衰减 |
|---|---|---|---|---|
| L0 | **事实记忆**（Fact） | LLM 从对话提取 | 语义记忆 | 慢（×0.95/天） |
| L1 | **经验记忆**（Experience） | 从多轮 Fact 归纳 | 情景记忆 | 中（×0.90/天） |
| L2 | **行为策略**（Strategy） | 用户手动设定/LLM 推导 | 程序性记忆 | 人工控制 |

### 4.2 Agent Experience 层（新增）

参考 **OpenViking 的 Trajectory→Experience 管道**：

```kotlin
// 新增实体
data class AgentExperience(
    @PrimaryKey val id: Long = 0,
    val situation: String,       // 场景描述
    val approach: String,        // 采取的方法
    val reflect: String,         // 反思总结
    val outcome: String,         // "success" / "failure"

```

经验记忆由 T+1 批量处理生成（不从对话实时提取），目的是减少 LLM 调用。

### 4.3 Meta-Memory 元记忆层（轻量版，参考 MetaMem）

元记忆是"如何使用记忆的策略"，由 LLM 在后台批量提炼：

```kotlin
data class MetaMemory(
    @PrimaryKey val id: Long = 0,
    val content: String,            // 元记忆内容（30词以内单句策略）
    val category: String,           // "retrieval" / "reasoning" / "conflict"
    val applyCount: Int = 0,        // 成功应用次数
    val confidence: Double = 0.5,   // 0~1 可信度
    val createdAt: Long,
    val updatedAt: Long
)
```

元记忆示例：
- "当记忆包含时间信息时，优先使用最新数据"
- "同一主题有矛盾记忆时，列出两个版本让用户选择"
- "用户的历史偏好查询比最近的单次表态更可靠"

元记忆注入 prompt 的格式：
```
[元记忆提示]
- 当记忆包含时间信息时，优先使用最新数据
- 同一主题有矛盾记忆时，列出两个版本让用户选择
```

**功耗**：元记忆由 T+1 批量生成，不在实时路径中产生 LLM 开销。

## 五、评分与融合排序

### 5.1 最终评分公式

参考 **mem0 的评分系统**（语义 + BM25 + 实体提升）：

```
final_score = 0.35 × fts_score + 0.25 × ppr_boost + 0.25 × recency + 0.15 × entity_boost

其中：
  fts_score: FTS rank(matchinfo) 归一化到 [0, 1]
  ppr_boost: PPR 传播分数，多个种子叠加后归一化
  recency: exp(-ln2/14 × idleDays)，14天半衰期
  entity_boost: 0.5 × (1 / (1 + 0.001 × (linkedCount-1)²))
               ↑ mem0 的公式，链接越多提升越小
```

**语义分数底线**：如果使用了嵌入引擎做语义检索，语义分数 < 0.3 的记忆直接被排除（即使 FTS 和实体提升很高也不召回）。

### 5.2 排序与 Token 预算

```kotlin
suspend fun retrieveRelevantMemories(
    userMessage: String,
    roleCardId: Long? = null,
    tokenBudget: Int = 1200    // 默认 1200 token 预算
): List<Memory> {
    // 1. PPR 检索 + 评分排序
    val candidates = pprRetrieve(userMessage, roleCardId)
    
    // 2. 按 final_score 降序
    val sorted = candidates.sortedByDescending { it.finalScore }
    
    // 3. 按 token 预算裁剪（优先取 L1 概览）
    var usedTokens = 0
    val result = mutableListOf<Memory>()
    for (candidate in sorted) {
        val text = candidate.l1Overview ?: candidate.content
        val tokens = estimateTokens(text)
        if (usedTokens + tokens > tokenBudget) break
        result.add(candidate)
        usedTokens += tokens
    }
    
    return result
}
```

## 八、功耗分析（Android 端）

| 操作 | 频率 | 计算位置 | 单次开销 | 日均功耗 |
|---|---|---|---|---|
| PPR 图检索 | 每次用户消息（约 100 次/天） | SQLite（本地） | < 5ms | 非常低 |
| FTS 检索 | 每次用户消息 | SQLite（本地） | < 2ms | 非常低 |
| 嵌入语义去重 | 每次 LLM 提取后（约 50 次/天） | Onnx（本地 NPU） | ~50ms | 低 |
| LLM 统一提取 | 每轮对话结束（约 50 次/天） | 云端 API | ~2s 网络 | 中 |
| 每日衰减调度 | 每日 1 次 | SQLite（本地） | < 10ms | 忽略不计 |
| T+1 批量处理 | 每日 1 次（空闲时段） | 云端 API | ~5s | 低（空闲时执行）|
| L0/L1 摘要生成 | 每轮对话结束（约 50 次/天） | 云端 LLM（与提取同一次调用）| 0（已包含在提取中）| 无额外 |

**关键优化点**：
1. **PPR 限制 1-hop**：不做多跳传播，避免 O(n²) 复杂度
2. **LLM 调用压缩**：一次调用输出记忆+链接+实体，而非三次独立调用
3. **T+1 批处理**：将经验和元记忆生成推迟到空闲时段
4. **纯本地检索**：图检索和 FTS 完全在 SQLite 完成，零网络请求
5. **Onnx 本地嵌入**：语义去重利用已有 NPU 加速，无需网络

## 附录：衰减曲线数学

```
给定记忆 M，当前强度 S(t) ∈ [0, 1]

每日衰减（参考艾宾浩斯遗忘曲线）：
  idleDays=1:  S *= 0.70   (遗忘 30%)
  idleDays=2:  S *= 0.80   (遗忘 20%)
  idleDays=3:  S *= 0.90   (遗忘 10%)
  idleDays≥4:  S *= 0.90/天 (每天遗忘 10%)

从 S=0.6 起（新建记忆初始值）：
  1 天未提及 → 0.6 × 0.70 = 0.420
  2 天未提及 → 0.420 × 0.80 = 0.336
  3 天未提及 → 0.336 × 0.90 = 0.302
  14 天后    → 0.302 × 0.90^11 ≈ 0.095 < 0.1 → 清理

强化规则：
  FTS 检索命中:    S += 0.05（上限 1.0）
  LLM 确认有价值:   S += 0.15（上限 1.0）
  用户手动编辑/收藏: S = 1.0

PPR 公式：
  记忆 M 检索得分 = 0.35 × FTS(matchinfo) + 0.25 × PPR(linkType, weight)
                    + 0.25 × exp(-ln2/14 × idleDays) + 0.15 × entityBoost

  entityBoost = 0.5 × (1 / (1 + 0.001 × (linkedCount-1)²))
```
