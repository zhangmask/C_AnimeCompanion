## 问题 A — 晋升机制断裂：短期记忆永不可能转为长期

### 位置

`MemoryRetriever.kt:49-52` + `MemoryDao.kt:74-75`

### 问题

`MemoryRetriever.retrieveRelevantMemories` 只对 **long_term** 层记忆递增引用计数：

```kotlin
// MemoryRetriever.kt:49-52
results
    .filter { it.layer == LONG_TERM_LAYER && it.referenceCount < MAX_REFERENCE_COUNT }
    .forEach { memoryDao.incrementReference(it.id) }
```

而晋升条件是 `referenceCount >= 5`（`MemoryDao.kt:74`）：

```sql
SELECT * FROM memories WHERE layer = 'short_term' AND referenceCount >= 5
```

**结果：** short_term 记忆的 `referenceCount` 永远是 0，永远达不到晋升门槛。**所有规则提取和模型提取的记忆（除非手动晋升）都会在 7 天后过期消失，永远无法变成 long_term。**

同样的，`MemoryPromptBuilder.buildLayered` / `retrieveRelevantWithScores` 中也不对 short_term 记忆递增引用计数。

### 影响

🔴 **严重** — 记忆系统几乎退化为一过性短期缓存。用户流失后重新对话，系统不记得之前学到的任何信息。

### 修复思路

`MemoryRetriever.retrieveRelevantMemories` 和 `MemoryPromptBuilder.retrieveRelevant` 中也应对 short_term 记忆递增引用计数，或另设晋升路径（如"被检索到 N 次 ± 存在时间 > M 天"）。

## 问题 B — `extractSelfDescription` 误抓闲聊作为人格特质

### 位置

`RuleBasedMemoryExtractor.kt:119-143`

### 问题

`extractSelfDescription` 中的 `matchEntire` 正则：

```kotlin
Regex("^我(比较|很|挺|有点)(.+)$")    // 匹配 "我比较喜欢这个回答"
Regex("^我是个(.+?)(的人)?$")          // 匹配 "我是个学生"
```

`normalizeClause` 会去除话语前缀，但保留"我"。以下日常表达会被提取为长期记忆：

| 用户说 | 提取的记忆 | 是否正确 |
|---|---|---|
| "我比较喜欢这个回答" | "用户比较喜欢这个回答" | ❌ 即兴评价，非稳定偏好 |
| "我觉得这个不错" → 去除"我觉得" → 不匹配 | 跳过 | ✅ |
| "我有点饿了" | "用户有点饿了" | ❌ 瞬时状态 |
| "我很满意" | "用户很满意" | ❌ 对话反馈 |
| "我是个学生" | "用户是个学生" | ✅ 稳定信息 |

### 影响

🟡 **中等** — 这是"随便说点就记了"的主要来源之一。用户随口评价或表达即时状态就被写入记忆库。

### 修复思路

增加置信度过滤：
1. 要求句子中含时间副词（"一般""通常""平时"）才视为稳定特质
2. 或"我比较/很/有点"只提取后续包含兴趣/性格关键词（"内向""开朗""喜欢XX"）的
3. 对短文本（< 8 字）不提取

## 问题 C — `extractPreference` 误抓即兴评价作为稳定偏好

### 位置

`RuleBasedMemoryExtractor.kt:55-67`

### 问题

```kotlin
Regex("^(?:我)?(?:很)?喜欢(.+)$")  // matchEntire
```

一个完整的子句如果恰好以"喜欢"开头，就会被记为偏好：

| 用户说 | 提取 | 是否正确 |
|---|---|---|
| "我喜欢这个游戏的画风" | "用户喜欢这个游戏的画风" | ❌ 当前对话的即兴评价 |
| "我喜欢吃辣的" | "用户喜欢吃辣的" | ✅ 稳定偏好 |
| "喜欢这个回答的风格" | "用户喜欢这个回答的风格" | ❌ 反馈而非偏好 |

### 影响

🟡 **中等** — 同上，是"乱记"的重要来源。

### 修复思路

- 添加上下文判断：如果"喜欢"的宾语包含"回答""这个""这个回答""你"等对话指代词，视为反馈不提取
- 或结合 MessageRole：只从用户消息的"自我介绍轮"中提取，而非每轮

## 问题 D — `extractResponseStyle` 将日常礼貌语记为回答偏好

### 位置

`RuleBasedMemoryExtractor.kt:146-172`

### 问题

```kotlin
Regex("^希望你(.+)$")               // "希望你能帮到我" → "用户偏好你能帮到我"
Regex("^(?:以后)?请(?:尽量)?(.+)$") // "请帮我查一下" → "用户偏好帮我查一下"
```

日常礼貌用语被错误提取为"回答偏好"：

| 用户说 | 提取 | 是否正确 |
|---|---|---|
| "希望你能帮到我" | "用户偏好你能帮到我" | ❌ 礼貌用语 |
| "请你帮我查一下" | "用户偏好帮我查一下" | ❌ 请求，非偏好 |
| "以后请直接一点" | "用户偏好直接一点" | ✅ 真实偏好 |

### 影响

🟡 **中等** — 用户简单说"请帮我XX"就被记录。

### 修复思路

- 检查宾语是否包含"帮"、"查"、"看看"等请求动词，如有则跳过
- 只保留含"回答"、"说话"、"语气"、"风格"、"简洁"、"详细"等风格关键词的

## 问题 E — `retrieveRelevantMemories` 有角色时忽略相关性

### 位置

`MemoryRepository.kt:44-51`

### 问题

```kotlin
suspend fun retrieveRelevantMemories(userMessage: String, roleCardId: Long? = null): List<Memory> {
    return if (roleCardId != null) {
        memoryDao.getPersistentMemoriesForRole(roleCardId)  // 返回 ALL
    } else {
        retriever.retrieveRelevantMemories(userMessage)       // FTS 搜索
    }
}
```

当 `roleCardId` 非空时，`userMessage` 参数被完全忽略，返回该角色的**全部** long_term 记忆，不经过 FTS 相关性匹配。

### 影响

🟡 **中等** — 有角色的对话会把该角色的全部长期记忆一股脑塞到 prompt 中，可能超过 token 预算，且对当前轮对话无用的记忆也会注入。

### 修复思路

有角色时也应走 FTS 检索（过滤 `roleCardId` 条件），而非全量返回。

## 问题 F — `promoteToLongTerm` 未清除 `expiresAt`

### 位置

`MemoryDao.kt:65-66`

### 问题

```sql
UPDATE memories SET layer = 'long_term', updatedAt = :now WHERE id = :id
```

晋升为长期记忆后，`expiresAt` 字段仍是旧的短期过期时间。下次 `cleanupExpiredShortTerm` 执行时（清理 `expiresAt < now` 的短期记忆），这条已晋升的记忆也会被清理，因为它的 `expiresAt` 没有置 null。

### 影响

🟢 **低**（当前晋升机制已断裂（问题A），实际不会触发此情况）

### 修复思路

```sql
UPDATE memories SET layer = 'long_term', expiresAt = NULL, updatedAt = :now WHERE id = :id
```

## 问题 G — `"定理|公式"` 知识过滤过于激进

### 位置

`RuleBasedMemoryExtractor.kt:257`

### 问题

```kotlin
Regex("勾股|定理|公式|定义|原理")
```

这个 `KNOWLEDGE_QUERY_PATTERNS` 条目是全局 `contains` 匹配，用户只要提到"定义""定理"等词，**整条消息都不会提取任何记忆**。

| 用户说 | 结果 | 是否正确 |
|---|---|---|
| "勾股定理是什么" | 跳过 | ✅ 确实是知识查询 |
| "我最近学了一个新公式" | 跳过全部 | ❌ 这是个人信息，应提取"最近学新公式" |
| "公式我记不太清了" | 跳过全部 | ❌ 可能包含可提取信息 |

### 影响

🟢 **低** — 误杀率不高，但可能丢失少量有价值信息。

### 修复思路

改用 `matchEntire` 或 `startsWith` 条件，而非全局 `contains`。

## 问题 H — FTS4 检索缺乏 `limit`，全量扫描开销大

### 位置

`MemoryDao.kt:59-60`

### 问题

FTS4 查询没有 `LIMIT` 子句，`searchByFTS` 会把所有匹配的记忆都返回，然后在 `MemoryRetriever.kt:47` 才 `.take(MAX_RESULTS = 5)`。当记忆库积累大量数据时，FTS 可能返回成百上千条然后丢弃大部分。

### 影响

🟢 **低**（当前记忆量小，不显著）

### 修复思路

FTS 查询直接加 `LIMIT 5`，避免数据库层→应用层的大量无效传输。

## 问题 I — `normalizeForDedup` 过度去除有效字符

### 位置

`MemoryRepository.kt:181-188`

### 问题

```kotlin
private val DEDUP_NOISE = listOf("打", "的", "了", "着", "过", "吧", "呢", "啊", "呀")
private fun normalizeForDedup(content: String): String {
    var result = content.trim().lowercase()
    for (noise in DEDUP_NOISE) {
        result = result.replace(noise, "")
    }
    return result
}
```

去除"打"后，"用户喜欢打篮球"和"用户喜欢篮球"去重碰撞，视为重复。但"打篮球"和"篮球"语义不同。

### 影响

🟢 **低** — 仅影响去重判断，去除"打"会导致少量有效信息被误判为重复。

### 修复思路

仅去除"的/了/吧/呢/啊/呀"这些纯语气词，去除"打"和"着"和"过"过于激进。

## 问题 J — MemoryPromptBuilder 中英文硬编码

### 位置

`MemoryPromptBuilder.kt:55,62,117-126,130`

### 问题

prompt 模板是中文硬编码，但系统支持英文 UI。记忆注入的提示词始终是中文（"从记忆中检索到的与当前对话相关的信息""以下内容均为用户本人的记忆"等），英文模式下也显示中文。

### 影响

🟢 **低** — 不直接影响功能，但英文用户会看到中英文混合的 system prompt。
